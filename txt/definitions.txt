Definitions

General
- config.json defines:
  - pair
  - active rung lifecycle
  - rung revisions
  - rung ranges by revision
  - rung allocations by revision
  - rung value snapshots by revision
  - borrow balances
  - borrow rates

- daily_metrics.json is the primary analysis dataset.
- Other uploaded analysis files may provide supporting summaries or alternate views.
- Only files uploaded in the current chat are in scope.
- A newer uploaded file overrides an older uploaded version of the same file.

Time scope
- Default analysis window starts at 2026-04-15.
- Data outside this window is excluded unless explicitly requested.

Trade (TR)
- A TR represents a single market trade event.

Rung lifecycle
- A rung is a persistent logical identity (for example A, B, C, D, S1, S2, S3).
- A rung may have one or more revisions.
- Each revision defines the parameters active for a specific time interval.
- effective_from = revision start timestamp
- effective_to = revision end timestamp
- effective_to = null means the revision is currently open-ended
- active at a given timestamp means:
  - TR.timestamp >= revision.effective_from
  - and (revision.effective_to is null or TR.timestamp < revision.effective_to)

Rung eligibility
- A rung revision is eligible if a revision is active at the TR timestamp.
- If no revision is active yet: not_started
- If a rung has been retired and no revision is active for the TR timestamp: inactive_for_period

Rung states
- participating: rung executed in TR
- skipped: rung was in range and eligible but did not execute while other eligible rungs did
- depleted: rung was in range and eligible but did not execute because liquidity was exhausted
- out_of_range: price outside the active revision range
- not_started: TR happened before the first revision effective_from
- inactive_for_period: TR happened after a rung revision ended and before any later revision began

Core metrics
- eligible_trs
- trades
- skipped_count
- out_of_range_count
- depleted_count

Derived metrics
- utilization_pct = trades / eligible_trs
- skipped_pct = skipped_count / eligible_trs
- out_of_range_pct = out_of_range_count / eligible_trs
- depleted_pct = depleted_count / eligible_trs
- fee_share_pct = fees / total fees
- capital_efficiency = fees / rung_value
- apy_gross = capital_efficiency * 365

Volume & fees
- Volume and fees are aggregated per rung.
- Fees are USD-equivalent.

Rung value
- rung_value refers to the value snapshot for the active rung revision.
- For historical analysis, rung_value should be taken from the revision active at the TR timestamp or reporting interval.
- portfolio_value = sum of values for currently active rung revisions

Buckets
- TRs may be grouped by price range.
- rung_counts track:
  - participating
  - skipped
  - depleted
  - out_of_range

Portfolio
- portfolio_value = sum of active rung values
- Borrow balances and borrow rates come from config.json

Predictions
- Based on realized fees.
- daily = current realized rate
- weekly = daily * 7
- monthly = daily * 30
- yearly = daily * 365
- APY derived from capital efficiency or yearly_income / rung_value, depending on the table context

Validation
- All eligible rungs must have a state.
- Only USDT/USDC data is included unless explicitly requested otherwise.
- No out-of-scope data unless requested.
- Revision timestamps are authoritative for historical eligibility.
- active flags in config are convenience fields and must not override revision timestamps.
